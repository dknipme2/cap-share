CAP · The Extensible Data Toolbox

Extensible by design, not by accident

The toolbox is how CAP separates what to do from how to do it. The architect picks a pattern; the operator picks the tool that implements it. Both live in files — swappable, portable, extendable without rewriting the system.

Two layers, two toolboxes

Design patterns, then tool implementations

CAP ships two paired toolboxes. The design toolbox (architect) is domain-agnostic and survives every swap. The tools toolbox (operator) is tech-specific and replaces wholesale when the compute platform changes.

The architect / operator boundary

Pattern vs. tool — two distinct decisions

Confusing these is a common failure mode. The architect declares what shape the data takes. The operator decides which tool on the current platform creates that shape efficiently. Either can push back to the other when the chosen pattern has no viable tool, or a preferred tool has no valid pattern.

Architect

Current toolbox contents

Five slots, each with a decision rule

The tools toolbox currently covers five implementation-time decisions. Each entry carries a When to apply trigger and a list of options with selection heuristics. Tools are options, not commandments — apply where context warrants.

SLOT 01 · COMPUTE RUNTIME

Databricks runtime selection

Which runtime for this computation? Ad-hoc and light ETL vs scheduled batch vs persistent interactive development. Cost discipline: warehouse auto-stops; FinOps estimate required for every write.

Serverless SQL Pro 2X-Small Serverless SQL Pro Medium/Large Job Cluster All-Purpose Cluster
SLOT 02 · PYTHON RUNTIME

pandas vs PySpark

Single-node data ≤ 1M rows and debugging iteration matters → pandas. Production pipeline, distributed, or > 1M rows → PySpark. Decision tree in file.

PySpark pandas
SLOT 03 · ORCHESTRATION

dbt vs raw SQL notebooks

Multi-analyst team, tests + docs co-located, complex dependency DAG → dbt. Single-analyst iteration, tight loop → raw SQL notebooks. CAP today defaults to raw SQL; dbt adoption triggers on team growth.

Raw SQL notebooks dbt
SLOT 04 · STORAGE

UC tables vs Volumes vs external

Structured data with schema → Unity Catalog Delta tables (default). Unstructured files (PDFs, JSON blobs, bundles) → Volumes. External data shared with non-Databricks consumers → External Locations.

UC Tables (Delta) UC Volumes External Locations
SLOT 05 · FILE FORMATS

Parquet · Delta · JSON

Delta for anything that needs MERGE, time travel, or schema evolution. Parquet for static exports to non-Delta consumers. JSON only for nested / semi-structured payloads where schema is genuinely dynamic.

Delta Parquet JSON

The extension model

How a new tool gets added — four steps, no compilation

The toolbox is markdown, not compiled code. Extension is append-only. Skill loaders re-read at invocation — new entries are live on the next session.

01 · TRIGGER
Real use case surfaces

An engagement needs a tool the toolbox doesn't cover. Evidence drives the add — not speculation.

02 · WRITE
Append a section

Same anatomy: When to apply, Selection rule, options, Pattern pairing. Same file.

03 · PAIR
Pair to a pattern

Link to the design pattern it implements. None yet? Surface to the architect.

04 · DISCOVER
Loader re-reads on invoke

Live on next session. No build, no deploy, no restart.

The runtime link · feasibility engine

How the toolbox ties back to the analytical question

The toolbox isn't decoupled from the analysis. The feasibility engine at tools/feasibility_check.py matches method requirements against data-source capabilities using a controlled vocabulary — the analytical role taxonomy. Pattern + tool + data-source capability + method requirement all meet here.

The analytical role taxonomy — 16 controlled-vocabulary roles (entity_identifier, outcome_measure, linkage_key, temporal_anchor, …) lives at knowledge/methodology/ANALYTICAL_ROLE_TAXONOMY.md. Both method YAMLs and data-source entity profiles cite from it. When analytical roles on the source match the roles the method needs, and the source's grain is at or below the method's minimum, feasibility is full. Missing a role or coarser-grained → partial or directional.

The underlying claim

Why this arrangement pays off

Extensibility is a design stance, not a feature.

The toolbox is a markdown file, not a plugin system. Extending it means appending a section; using it means a skill loader re-reads the file at invocation. No new abstractions, no registration, no re-deploy. The contract is the file's anatomy — a When to apply trigger, a selection rule, a pairing note.

The architect / operator split is load-bearing. Keeping pattern and tool in separate files with different lifecycles lets each evolve on its own schedule. Patterns accrete over years across the industry; tools turn over every few years with platform shifts. Coupling them would force rewrites on every tool swap.

The feasibility engine is the runtime proof. The taxonomy isn't decorative — it's the interface between a method's data requirements and a source's capabilities. When a new data source profile lands, feasibility re-scores automatically. When a new method YAML lands, its data requirements are checked against every source the graph knows about. The toolbox is the discovery surface; the feasibility engine is the runtime match.
